# Tarot Telegram Bot

Рабочий README для разработки Telegram-бота и Telegram Mini App с раскладами Таро. Документ собран на основе плана разработки из PDF и дополнен архитектурными уточнениями.

## Идея продукта

Пользователь запускает Telegram-бота, выбирает расклад, открывает Mini App, видит закрытые карты, выбирает любые нажатием, карты красиво переворачиваются, а затем показывается трактовка.

Главная особенность продукта: расклад выглядит интерактивным и персональным, но технически формируется заранее на backend как стабильный результат. Mini App не генерирует карты, а только раскрывает уже созданный расклад через визуальный выбор.

## Ключевой принцип формирования расклада

Расклад формируется заранее на backend как неизменяемый снимок. Выбор закрытых карт в Mini App является интерактивным способом раскрытия уже созданного расклада: пользователь выбирает любые визуальные слоты, а система последовательно открывает карты, заранее назначенные позициям расклада.

Это означает:

- Backend создает `reading` в момент старта расклада.
- Карты, позиции, `score`, `tags`, сочетания и итоговая трактовка фиксируются сразу.
- Mini App не генерирует карты и не влияет на результат.
- Пользователь видит 12-15 закрытых карт и выбирает любые нажатием.
- Каждый выбранный слот открывает следующую карту из заранее созданного `reading`.
- При перезагрузке Mini App выбранные слоты и открытые карты восстанавливаются.
- Старые расклады показываются из `result_snapshot_jsonb` и не меняются после обновления трактовок.

## Модель состояния reading

У `readings` должна быть явная статусная модель:

```text
created    - reading создан, карты уже выбраны, пользователь еще не начал раскрытие
selecting  - пользователь выбирает закрытые слоты
revealed   - все карты раскрыты, результат показан
expired    - активный период расклада закончился
cancelled  - опциональный статус для отмененных или ошибочных сценариев
```

Рекомендуемые поля `readings`:

```text
id
user_id
spread_id
status
cards_jsonb
positions_jsonb
selected_slots_jsonb
revealed_count
tags_jsonb
score
result_snapshot_jsonb
share_image_url
created_at
expires_at
```

## Безопасность Telegram Mini App

Backend должен проверять Telegram WebApp `initData` на каждом запросе Mini App:

- проверять HMAC-подпись;
- проверять `auth_date`;
- ограничивать срок жизни `initData`;
- получать `telegram_id` только из проверенного `initData`;
- не доверять `telegram_id`, переданному отдельно с frontend.

## Оплата

Telegram Payments используется как первый платежный провайдер. Архитектура должна позволять позже добавить ЮKassa.

Важные правила:

- не брать повторную оплату за уже активный расклад;
- не создавать платный reading до подтверждения успешной оплаты;
- повторные payment callback должны быть идемпотентными;
- `payment` должен быть связан с `reading_id`;
- у платежей должны быть статусы `pending`, `paid`, `failed`, `refunded`.

## Бесплатный лимит

Для правила "1 бесплатный расклад в сутки" лучше использовать отдельную таблицу:

```text
free_usage
id
user_id
usage_date
reading_id
created_at
```

Нужен уникальный индекс на `(user_id, usage_date)`, чтобы повторные клики не создавали обход лимита.

## Контент и юридическая аккуратность

Тексты трактовок должны быть:

- короткими;
- спокойными;
- нейтральными;
- не пугающими;
- без обещаний точного будущего;
- без медицинских, юридических и финансовых советов;
- без дословного копирования из источников.

Рекомендуется добавить `interpretation_set_code` или `content_version`, чтобы понимать, какой набор трактовок использовался при создании старого расклада.

## Как проверять проект на ранних стадиях

### После создания каркаса

Проверь, что сервисы стартуют:

```text
docker compose up --build
```

Минимум должен работать:

```text
GET /health -> {"status":"ok"}
```

### После подключения БД

Проверь миграции:

```text
alembic upgrade head
alembic current
```

Проверь таблицы в PostgreSQL и базовый insert/select через SQLAlchemy.

### После seed-данных

Проверь количество записей:

```text
spreads: ожидаем все расклады MVP
tarot_cards: ожидаем 78 карт
card_interpretations: ожидаем трактовки для каждой карты
```

### После движка раскладов

Сделай простой backend smoke test:

```text
create_or_get_reading(test_user_id, "card_of_day")
```

Проверь:

- создается `reading`;
- `cards_jsonb` заполнен;
- `result_snapshot_jsonb` заполнен;
- повторный вызов возвращает тот же `reading_id`;
- после истечения `expires_at` создается новый reading.

### После API

Проверь запросы через Swagger/FastAPI docs или curl:

```text
GET /api/spreads
POST /api/readings/{spread_code}/start
POST /api/readings/{reading_id}/select-cards
GET /api/readings/{reading_id}
```

Для dev-режима можно временно использовать mock initData, но в production проверка Telegram initData обязательна.

### После Mini App

Проверь в браузере:

```text
npm run dev
```

Что смотреть:

- экран помещается в мобильный viewport;
- закрытые карты не перекрывают друг друга;
- выбранная карта переворачивается;
- открывается карта из backend, а не случайная на frontend;
- после перезагрузки выбранные слоты сохраняются;
- после последней карты появляется трактовка.

### После бота

Проверь вручную в Telegram:

```text
/start
выбор категории
выбор расклада
открытие Mini App
```

В БД должен появиться или обновиться пользователь.

### После оплаты

Проверяй только в тестовом/sandbox-сценарии:

- pending payment создается до оплаты;
- paid payment создается после успешной оплаты;
- reading создается только после successful payment;
- повторный callback не создает дубликаты.

## Минимальный набор автотестов

На ранних стадиях достаточно таких тестов:

- миграции применяются на пустую БД;
- seed создает 78 карт;
- у каждого расклада количество позиций равно `cards_count`;
- `create_or_get_reading` идемпотентен до `expires_at`;
- `select-cards` не принимает повторный slot_index;
- `select-cards` не позволяет открыть больше карт, чем нужно;
- старый `result_snapshot_jsonb` не меняется после изменения трактовки;
- бесплатный лимит срабатывает один раз в сутки;
- payment callback идемпотентен.

## Приоритет MVP из исходного плана

Обязательно до запуска:

1. База данных
2. Telegram-бот
3. Каталог раскладов
4. База 78 карт
5. Краткие трактовки
6. Движок раскладов
7. Mini App
8. Правило неизменности расклада
9. Бесплатный расклад
10. Оплата
11. История
12. Поделиться

Можно после запуска:

- расширенная админка;
- больше сочетаний карт;
- перевернутые карты;
- несколько колод;
- дополнительные темы оформления;
- обновление трактовки старого расклада.

Не делаем в MVP:

- годовой прогноз;
- музыка;
- чат с тарологом;
- гороскопы;
- форум;
- уровни пользователей.
